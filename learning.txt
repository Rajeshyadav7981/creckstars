  Where to go IF you grew
  
  If hypothetically traffic exploded to that scale, the cheapest path that scales linearly is:
  
  Pattern A — Horizontal scale + sticky sessions (good up to ~250k WS)
  
                      ┌──────────────────────┐
                      │  Nginx / HAProxy     │  ← sticky on match_id
                      │  (consistent hash)   │
                      └──┬───────┬──────┬────┘
                         │       │      │
                    ┌────▼─┐  ┌──▼──┐  ┌▼────┐
                    │ API+ │  │ API+│  │ API+│   ← each does its own WS
                    │ WS 1 │  │ WS 2│  │ WS 3│
                    └────┬─┘  └──┬──┘  └─┬───┘
                         │       │       │
                         └──┐  ┌─┴─┐  ┌──┘
                            ▼  ▼   ▼  ▼
                         ┌──────────────┐
                         │ Redis Pub/Sub│  ← single cluster
                         │  + PG primary│
                         └──────────────┘

  Sticky hashing on match_id means every viewer of match X lands on the same VM → broadcast stays
  local, no Redis Pub/Sub hop. This single architectural change increases WS capacity ~5× because
  you eliminate the cross-worker fanout entirely for the common case.

  The code change is small — the LB does the routing; the backend doesn't even know.
  
  Pattern B — Dedicated WS tier (good up to ~1M WS)

  Split the deploy:
  - API VMs: handle /score, /scorecard, etc. — pure HTTP
  - WS VMs: handle ONLY wss:// connections, subscribe to Redis Pub/Sub
  - Redis cluster: 3-node minimum, sharded by match_id

  A score lands on an API VM → writes DB → publishes to Redis. WS VMs subscribe and fan out to
  their local clients. Each WS VM holds ~50k connections.

  For 1M WS you'd need ~20 WS VMs. Each VM is cheap because it's pure I/O (Python + uvloop is
  fine).    

  Pattern C — Managed edge service (only for truly viral)
  
  Cloudflare Durable Objects, Ably, Pusher, or PubNub. You publish events to them; they fan out to
   viewers connected to edge PoPs near them. No connection ever reaches your origin.

  Cost is per-message, so it's only economical for very broad fan-out (think IPL final, 5M
  viewers).

  My honest recommendation

  For community cricket on a 2 GB VM, the current setup serving the realistic 5–10k peak is the 
  right answer. You don't need any of the tiers above today.

  The single most useful thing you can do before scaling out is make broadcast cheaper per 
  connection:
  - ✅ uvloop (shipped)
  - ✅ orjson on WS path (shipped)
  - ⏳ When/if you outgrow one VM, add sticky-hash routing on match_id at the load balancer.
  That's a 10× free multiplier before you need a dedicated WS tier.
 
  Numbers worth memorizing for capacity planning:

  Single 2 GB VM, current code:                  ~5–10k total WS
  Single 8 GB VM, current code:                  ~30–50k total WS
  3-VM cluster, sticky on match_id:              ~150k total WS
  Above that → dedicated WS tier (split deploy)

  TL;DR: 1k matches × 1k viewers = 1M WS is a "managed-edge-service" scale problem, not a
  code-tuning problem. Your current architecture is right-sized for community cricket and has
  clear next steps if it explodes.
